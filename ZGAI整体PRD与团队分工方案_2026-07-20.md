# ZGAI 整体 PRD 与团队分工方案

> 项目：zgai-至高律所管理系统  
> 日期：2026-07-20  
> 最近更新：2026-07-21  
> 用途：用于领导汇报、外部团队委托、Codex 多任务拆分、AI 接入测试规划  
> 当前策略：先稳定案件/客户/审批/文档/财务核心闭环，再推进批量导入、知识库与 AI

---

## 1. 项目总目标

ZGAI 的目标不是做一个泛 OA 系统，而是做至高律所内部的“案件与客户数据底座”。系统应先支撑真实试用，再逐步叠加 AI。

核心闭环：

1. 客户进入系统；
2. 发起利冲检查；
3. 律师发起立案申请；
4. 行政管理进行利冲审查和立案初审；
5. 主任或授权管理人员进行终审或监管；
6. 审批通过后自动形成一案一档；
7. 案件办理过程中持续沉淀文档、时间线、财务、发票、办案记录；
8. 后续通过 RAG 与 AI 辅助录案、检索、文书生成、风险提示。

当前阶段的判断：

- 已经有核心业务雏形；
- 还没有达到“长期稳定多人使用”的程度；
- 下一步重点不是继续铺新模块，而是稳定数据底座、权限边界、审批闭环、文档归档与批量导入。

## 2. 当前系统状态摘要

### 2.1 已具备能力

| 模块 | 当前状态 | 判断 |
|---|---|---|
| 登录与账号 | 已具备 | 可继续完善角色权限 |
| 部门与身份类别 | 已具备 | 已有九大部门、主任/行政/律师等身份基础 |
| 客户管理 | 基础可用 | 需继续完善主体库、权限、利冲记录 |
| 案件管理 | 基础可用 | 单案立案、列表、详情已具备，但专业化仍不足 |
| 立案审批 | 基础可用 | 需强化行政/主任流程、弹窗资料、消息一致性 |
| 案件文件 | 已开始 | NAS 路径已接入，需标准化目录、版本、元数据 |
| 财务发票 | 已开放基础页 | 需稳定申请、反馈、完成锁定闭环 |
| AI 配置 | 已有雏形 | 当前已有 DeepSeek/Ollama/Qdrant 相关配置与服务 |
| RAG 知识库 | 已有基础 | 需限定先索引全所知识和公共模板 |

### 2.2 主要缺口

| 缺口 | 影响 | 优先级 |
|---|---|---|
| PostgreSQL 正式主库未完全落地 | H2 不适合多人长期测试 | P0 |
| 权限边界还不够细 | 普通律师/行政/主任看到的数据需严格区分 | P0 |
| 审批中心和消息中心仍需完全一致 | 审批体验会影响试用信任度 | P0 |
| 案件文件库还缺标准目录和版本管理 | 后期材料多时会失控 | P0 |
| 旧系统资料检索接入未实现 | 后续需要按现有案件要素检索旧案件对应材料 | P1 |
| ssb 项目尚未接入 | 银行类诉讼已有独立能力，ZGAI 先做入口跳转，后续融合 | P1 |
| AI 测试缺少安全边界和评价集 | 容易过早接入敏感材料 | P1 |

### 2.3 第一阶段必须真实可用的范围

第一阶段不是概念 Demo，而是要让律所内部人员可以拿真实业务流程进行试用。必须可用的模块如下：

| 模块 | 第一阶段目标 | 验收口径 |
|---|---|---|
| 案件管理 | 建立案件台账、查询、筛选、详情、状态流转 | 普通律师只能看自己相关案件，部门主管看本部门案件，行政/主任按权限看整体信息 |
| 客户管理 | 建立客户主体库、案源人、承办人、所属部门 | 普通律师只看自己作为案源人或承办人的客户，利冲检查仍可检索全客户库 |
| 立案审批 | 律师发起，行政先审，主任终审或监管 | 审批中心和消息中心一致，审批资料可在右侧抽屉查看，同意/驳回清晰 |
| 案件文件/NAS | 一案一档，一档多目录 | 审批通过后生成标准目录，上传文件有元数据、路径、版本和操作记录 |
| 发票申请 | 当前闭环保持稳定 | 申请、修改、删除、反馈、完成锁定可按既有规则运行 |
| AI知识库 | 独立入口和工作台入口 | 只放法律法规、律所内部制度、公共模板、参考知识，不放真实案件私密材料 |
| 旧系统资料检索 | 不做全量导入，先做资料检索入口 | 基于 ZGAI 现有案件/客户要素检索旧案件对应资料，并保留查询记录 |

可以先试验但不纳入第一阶段硬性交付的模块：

| 模块 | 当前策略 |
|---|---|
| AI 文书生成 | 先用脱敏材料和公共模板生成草稿，正式文书仍需人工确认 |
| 银行批量建案 | 因已有 ssb 项目，ZGAI 第一阶段只做直接跳转入口，后续再融合接口或数据 |

### 2.4 2026-07-21 需求确认基线

本节作为后续研发、外包、验收的决策基线。未重新确认前，开发按以下口径执行：

| 事项 | 确认口径 |
|---|---|
| 旧系统导入 | 暂不处理全量导入，待资料整理后再做 |
| 旧系统资料 | 当前先做“基于现有案件/客户要素检索旧案件资料”，不是直接迁移旧库 |
| AI 知识库首批内容 | 法律法规、律所内部制度、公共模板、参考 Alpha 类法律知识组织方式 |
| AI 知识库入口 | 左侧导航栏需要独立入口，主工作台也需要快捷入口 |
| 汇报材料 | 先完成当前功能推进，再考虑再次制作汇报稿 |
| 行政权限 | 行政管理可以查看案件整体信息，用于审批、利冲和案卷管理 |
| 主任权限 | 主任拥有全权限 |
| 发票模块 | 当前视为已完成，后续有升级需求时再安排 |
| UI 标准 | 采用前文提出的 macOS 式简洁、专业、低噪声标准 |
| 多团队研发 | 可以引入外部团队，也可以用多个 Codex 账号/任务并行，但必须按功能域和分支拆分 |
| 银行批量建案 | 第一阶段先从 ZGAI 直接跳转 ssb，后续再把 ssb 融入系统 |
| 第一阶段验收 | 以真实可用为准，优先验证案件台账、客户库、立案审批、文档归档、财务开票 |

## 3. 任务分工总览

建议把工作分成三类：

1. 外部团队：做边界清晰、交付物明确、需要专业设计/部署/安全经验的工作。
2. Codex：做当前代码库内可持续迭代、可快速验证、可自动化测试的工作。
3. 现在接入 AI 测试：只做低风险、可控、可评估的 AI 能力，不直接处理全量私密案件材料。

## 4. 建议委托外部团队实现的内容

### 4.1 UI/UX 与设计系统

委托原因：

- 你已经明确希望推倒当前 UI 风格，转向类似 macOS 的简洁、专业、低噪声界面；
- 这类工作需要系统性视觉规范，不适合只靠零散改页面；
- 外部设计团队可交付 Figma、组件规范、页面原型，内部/Codex 再实现。

外部交付物：

- ZGAI 设计系统：颜色、字体、间距、表格、表单、按钮、弹窗、抽屉、导航、状态标签；
- 律师工作台原型：以日历、案件、待办、最近文档为核心；
- 行政工作台原型：以审批、消息、利冲、案卷建档为核心；
- 案件详情页信息架构；
- 客户详情页信息架构；
- 审批中心与消息中心交互稿；
- 批量导入向导交互稿。

验收标准：

- 原型覆盖桌面端主流程；
- 普通律师、行政管理、主任、财务至少四类角色有差异化界面；
- 每个页面明确主操作、次操作、危险操作；
- 交付可让前端直接照图实现。

### 4.2 数据库与生产部署架构评审

委托原因：

- PostgreSQL、NAS、Qdrant、备份、安全访问属于长期基础设施；
- 试用阶段可以由 Codex 实现，但正式上线前需要外部有经验团队做架构审查；
- 律所数据敏感，备份、权限、日志、灾备不能靠临时脚本长期支撑。

外部交付物：

- PostgreSQL 部署方案；
- 数据库备份与恢复方案；
- NAS 案件文件库目录规范；
- 局域网访问与公网临时访问安全方案；
- HTTPS、反向代理、域名、证书方案；
- 日志留存与异常排查规范；
- 数据库迁移和回滚方案。

验收标准：

- 可从备份恢复数据库；
- 案件文件和数据库元数据可以对账；
- 权限、账号、端口、远程访问有明确安全边界；
- 形成部署文档。

### 4.3 旧系统资料梳理与检索接入协助

委托原因：

- 旧系统导入暂不作为当前开发重点；
- 后续目标不是立即全量迁移，而是基于 ZGAI 现有案件/客户要素，检索旧案件对应资料；
- 旧系统字段、旧案号、客户名称、承办人、文件路径往往不规范，需要先梳理查询规则和资料索引。

外部交付物：

- 旧系统案件资料字段字典；
- 旧系统资料检索入口或导出方式说明；
- 案件名称、客户名称、旧案号、承办人、部门、时间等检索字段映射；
- 历史文件路径清单；
- 10-20 行脱敏样例；
- 旧资料检索规则说明。

验收标准：

- Codex 可以据此实现“旧资料检索入口”；
- 能从 ZGAI 案件/客户页面跳转或查询到旧系统对应资料；
- 每次查询有来源、时间、操作人记录。

### 4.4 法律模板与制度文本专业化

委托原因：

- 利冲报告、委托合同、授权委托书、风险告知、立案审批表等不仅是技术模板，也是律所管理制度；
- Codex 可以实现模板生成，但模板内容应由律所内部或外部专业顾问确认。

外部或律所内部交付物：

- 利冲审查报告模板；
- 立案审批表模板；
- 结案归档表模板；
- 常用委托合同模板；
- 发票申请与财务审批规则；
- 案卷目录标准。

验收标准：

- 模板内容经主任或管理层确认；
- 系统生成文档只负责套版，不擅自改变制度口径。

### 4.5 安全与合规评估

委托原因：

- AI 接入、客户信息、案件材料、员工账号都涉及敏感数据；
- 正式上线前应评估数据出境、云端模型调用、账号权限、日志脱敏。

外部交付物：

- 敏感数据分级；
- AI 调用合规规则；
- 云端模型可用范围；
- 本地模型部署建议；
- 账号权限审计规则；
- 安全测试报告。

验收标准：

- 明确哪些数据不能传云端；
- 明确谁能导出客户/案件/文件；
- 明确日志保留和脱敏策略。

## 5. 建议由 Codex 持续实现的内容

Codex 适合处理“当前代码库内、需求明确、可以快速验证”的任务。建议将 Codex 作为内部快速研发与维护主力。

### 5.1 P0：核心闭环稳定化

Codex 可直接实现：

- 修复案件列表批量操作参数问题；
- 审批中心与消息中心数据一致；
- 查看审批改为右侧抽屉，不跳转；
- 行政审批按钮明确为“同意立案/驳回立案”；
- 驳回理由回传发起人；
- admin 查看全部审批；
- 田颖思等授权人员编辑归档前案件；
- 律师/行政/主任不同首页与菜单；
- 案件列表增加或完善案由、部门筛选；
- 客户列表按案源人、承办人、所属部门控制可见范围；
- 指定人员全客户查看权限；
- 发票申请闭环修复。

验收标准：

- 使用普通律师、行政、主任、财务、admin 五类账号完整跑通；
- 前端 build 通过；
- 后端测试或打包通过；
- 每次改动后保留明确测试记录。

### 5.2 P1：数据库与文档底座

Codex 可实现：

- PostgreSQL profile 完整化；
- 核心 schema 补齐；
- `case_document`、`document_folder`、文档版本字段；
- 审批通过后创建标准 NAS 目录；
- 上传文件保存元数据和实际路径；
- 文件版本不覆盖；
- 文档索引状态字段；
- 基础备份脚本。

需外部或你确认后再做：

- 正式生产部署拓扑；
- 备份保留周期；
- NAS 权限账号；
- 是否启用公网访问。

### 5.3 P1：旧系统资料检索与 ssb 项目接入

Codex 可实现：

- 旧系统资料检索入口；
- 基于案件名称、客户名称、案号、承办人等字段查询旧资料；
- ZGAI 案件详情页关联旧资料链接；
- 查询留痕；
- ssb 项目菜单入口；
- 从 ZGAI 直接跳转 ssb；
- 后续预留 ssb 数据融合接口。

实现原则：

- 旧系统暂不做全量导入；
- 导入问题先理解为“用现有 ZGAI 要素检索旧案资料”；
- 检索字段优先采用案件名称、客户名称、旧案号、承办人、部门、时间；
- 查询结果应显示资料来源、匹配原因、旧系统路径或跳转地址；
- 每次查询要记录操作人、时间、关键词、命中结果；
- ssb 第一阶段采用独立入口直接跳转，后续再判断单点登录、API、数据库视图或深度融合。

需要你或外部提供：

- 旧系统资料访问方式；
- 旧系统可用检索字段；
- ssb 项目的访问地址、登录方式、是否支持单点登录；
- 后续 ssb 是否能提供 API 或数据库视图。

### 5.4 P1：测试与验收自动化

Codex 可实现：

- 后端核心接口测试；
- 前端关键流程 Playwright 测试；
- 登录角色测试；
- 审批流程测试；
- 发票流程测试；
- 导入流程测试；
- 构建脚本和一键启动脚本；
- 测试报告 Markdown/PDF。

### 5.5 P2：AI 网关与 RAG MVP

Codex 可实现：

- AI 配置面板完善；
- OpenAI-compatible API 适配；
- DeepSeek/通义/本地 Ollama 配置测试；
- AI 调用日志；
- AI 使用权限；
- 公共知识库 RAG；
- 公共模板索引；
- 文档摘要与标签建议；
- 案件信息抽取草稿；
- 利冲辅助提示。

不建议 Codex 单独决定：

- 哪些案件材料可以上传云端；
- AI 生成法律意见是否可直接给客户；
- 是否自动索引全部案件文件。

## 6. 现在需要接入 AI 测试的内容

AI 现在应该接入，但必须控制范围。建议先测“辅助录入和内部效率”，不要先测“自动办案结论”。

### 6.1 现在就该测试

| AI 场景 | 是否现在测试 | 原因 |
|---|---|---|
| AI 配置连通性 | 是 | 当前已有 AIConfig，可先验证模型供应商 |
| 非敏感法律问答 | 是 | 可用公开法规、模板测试 |
| 公共知识库 RAG | 是 | 风险低，能验证检索链路 |
| 公共模板检索 | 是 | 对文书生成有直接价值 |
| 案件信息抽取草稿 | 是，但用脱敏材料 | 可辅助新建立案 |
| 文档自动分类 | 是，但先用样本文档 | 可服务案卷归档 |
| 利冲辅助提示 | 是，但只作提示 | 最终结论仍由行政管理确认 |
| 发票申请字段抽取 | 是 | 风险低，能提升录入效率 |

知识库首批内容范围：

- 法律法规、司法解释、公开规范性文件；
- 律所内部制度、办案流程、审批规则、归档规则；
- 公共模板，如合同、函件、申请书、报告模板；
- 参考 alpha 法律系统的知识组织方式，但不直接依赖其闭源能力；
- 后续经人工审核后的案件沉淀材料。

知识库首批禁止内容：

- 未脱敏真实案件材料；
- 客户身份证件、联系方式、合同原件、证据原件；
- 未经授权的旧系统全量案卷；
- 尚未确认可对 AI 开放的律所敏感制度。

### 6.2 暂不建议测试

| AI 场景 | 暂缓原因 |
|---|---|
| 全量案件文件自动进 RAG | 权限和保密边界未完成 |
| 客户私密材料上传云端模型 | 合规规则未确认 |
| AI 自动给出最终利冲结论 | 需要人工审查负责 |
| AI 自动生成正式法律意见 | 风险高，应先作为草稿 |
| 多台 Mac mini 串联跑大模型 | 工程复杂度高，不适合当前阶段 |
| 32B/35B 本地模型在当前 Mac mini 16GB 上生产使用 | 内存不足，不稳定 |

### 6.3 AI 测试优先顺序

第一批测试：

1. 云端 AI 配置连通性；
2. AI 调用日志；
3. 独立 AI 知识库页面；

## 7. AI 接入技术判断

### 7.1 当前系统已有基础

代码中已经存在：

- `AIConfigService`；
- `AIConfigController`；
- `AIDocumentService`；
- `AiChatService`；
- `RAGKnowledgeService`；
- `QdrantVectorService`；
- DeepSeek 配置；
- Ollama 本地配置；
- DashScope embedding 配置；
- Qdrant docker-compose。

因此，不需要从零开始接入 AI。更合理的做法是先把现有 AI 能力整理成“可配置、可测试、可记录、可禁用”的 AI 网关。

### 7.2 云端优先策略

由于当前硬件是 Mac mini M4 / 16GB，适合作为开发与 Demo 节点，不适合承担 32B/35B 级本地大模型生产推理。

短期建议：

- 云端 AI 处理非敏感或脱敏内容；
- 本地只做轻量模型验证；
- 所有 AI 调用写日志；
- 管理员可开关 AI 功能；
- 敏感案件材料默认禁止传云端。

### 7.3 本地模型策略

Qwen3-32B GGUF 官方/社区模型已有量化版本，Q4_K_M 文件约 19.8GB，Q8_0 约 34.8GB。它适合在高内存 Mac Studio 上测试，不适合在当前 16GB Mac mini 上稳定运行。

本地 AI 分三档：

| 档位 | 硬件 | 作用 |
|---|---|---|
| 当前最低成本 | 现有 Mac mini M4 / 16GB | 云端 AI + 小模型连通测试 |
| 低成本本地 AI | Mac Studio M4 Max / 64GB 或 128GB | Qwen3-32B Q4/Q5 低并发测试 |
| 正式本地 AI | Mac Studio M3 Ultra / 256GB 起 | 私密案件材料本地处理、较稳定多人使用 |

参考：

- Qwen3-32B GGUF：https://huggingface.co/Qwen/Qwen3-32B-GGUF
- Mac mini M4 技术规格：https://www.apple.com/mac-mini/specs/
- Mac Studio 技术规格：https://www.apple.com.cn/mac-studio/specs/

### 9.5 多 Codex 并行开发方式

可以启用多个 Codex 账号或多个 Codex 任务同步推进，但必须按“功能域 + 分支 + 接口契约”拆开，不能让多个 Codex 同时改同一批核心文件。

建议并行分组：

| Codex/团队 | 负责范围 | 分支建议 | 禁止事项 |
|---|---|---|---|
| Codex-A 核心后端 | 案件、客户、审批、权限、PostgreSQL | `feature/core-backend` | 不改 UI 大布局 |
| Codex-B 前端体验 | 工作台、导航、案件/客户/审批页面 | `feature/frontend-workbench` | 不改后端实体结构 |
| Codex-C 文档/NAS | 案件目录、文件上传、版本、旧资料检索入口 | `feature/document-nas` | 不改审批流程 |
| Codex-D AI/RAG | AI 配置、知识库入口、法规/制度 RAG | `feature/ai-rag` | 不处理真实案件材料 |
| Codex-E 测试/运维 | 启动脚本、测试脚本、部署文档、回归清单 | `feature/test-devops` | 不改业务逻辑 |

协作规则：

- 每个 Codex 只处理一个功能域；
- 每个功能域先写接口契约，再开始改代码；
- 每日只合并一次到主分支；
- 合并前必须跑前端 build、后端打包或对应核心测试；
- 涉及数据库字段变更时必须先在 PRD/迁移说明中登记；
- 涉及权限、审批、客户、案件主表的改动必须由一个“集成负责人”统一审查；
- 外部团队如果参与代码开发，也必须遵守同样的分支和接口契约规则。

现实判断：

- 多 Codex 能提高并行速度，尤其适合页面、文档、测试、AI、脚本并行；
- 对案件/客户/审批这类核心链路，不建议多人同时无边界改动；
- 最佳模式是：Codex 做主开发，外部团队做架构/部署/UI/安全，少量功能开发由外部团队按接口契约接入。

### 9.6 下一阶段建议拆成 6 个并行任务

若启用多个 Codex 或外部团队，建议从以下 6 个任务开始拆分，避免互相踩文件：

| 任务 | 负责人建议 | 目标 | 主要交付 |
|---|---|---|---|
| A. 审批闭环专项 | Codex 或后端小队 | 立案审批、消息中心、审批中心完全一致 | 审批抽屉、同意/驳回、驳回理由、审批记录、admin/行政/主任权限 |
| B. 案件与客户权限专项 | Codex | 数据可见范围稳定 | 案件部门权限、客户案源人/承办人权限、主任全权限、行政查看整体案件信息 |
| C. 案件文件/NAS 专项 | Codex + 运维 | 一案一档可用 | NAS 目录模板、文件上传、版本、元数据、文件检索 |
| D. AI知识库专项 | Codex | 建立低风险 RAG MVP | 法规/制度/模板入库、索引状态、问答入口、调用日志 |
| E. UI/UX 重构专项 | 外部设计团队优先 | macOS 式专业简洁界面 | Figma、设计系统、律师/行政/主任/财务工作台原型 |
| F. 旧资料检索与 ssb 接入专项 | 外部梳理 + Codex 实现 | 旧资料可查，ssb 可进入 | 检索字段映射、旧资料入口、ssb 跳转配置、查询留痕 |

集成规则：

- A、B 属于核心权限与审批，必须由同一集成负责人最终审查；
- C 不应改审批业务规则，只消费案件状态和案件编号；
- D 第一阶段不得接触真实案件文件；
- E 先交付原型和组件规范，不直接改生产代码也可以；
- F 第一阶段以“跳转和检索”为目标，不做深度数据融合。
