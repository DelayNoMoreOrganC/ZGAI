# ZGAI NAS 部署、AI 接入与预算方案

版本：1.2
日期：2026-07-23
阶段：采购与生产部署前方案

## 1. 已确认决策

1. ZGAI 核心系统最终部署在 NAS 容器中。
2. 旧案资料需要导入，先由行政人员使用标准 Excel 模板整理，再做预检、试导入和正式导入。
3. AI 采用“本地优先、云端手动选择”：本地通过 LM Studio/llama.cpp 调用 Qwen；云端接入 DeepSeek、GLM、Kimi。
4. 云端 API Token 为律所统一配置，由后端保管，浏览器和日志不得取得明文。
5. Word、Excel 在服务端生成；需要 PDF 预览或格式转换时使用 LibreOffice headless 容器，不依赖生产设备安装 Microsoft Office。
6. 生产核心、AI 推理和备份分离：NAS 承载业务，模型机承载推理，独立介质承载备份。

## 2. 目标架构

```text
律师/行政浏览器
      |
   HTTPS/反向代理
      |
NAS Container Manager
  |- ZGAI 前端
  |- ZGAI 后端
  |- PostgreSQL 16（NAS SSD 本地卷）
  |- Qdrant（NAS SSD 本地卷）
  |- LibreOffice headless（文档转 PDF/预览）
  `- 定时备份任务
      |
      +-- NAS 案件文件卷（HDD/SSD 阵列，一案一档）
      +-- 独立备份卷/外置盘（不与生产卷共命运）
      `-- 局域网模型机（LM Studio/llama.cpp）
             `-- qwen/qwen3.6-35b-a3b（模型机已核验）

后端 AI 网关 --手动选择--> DeepSeek / GLM / Kimi
```

### 2.1 NAS 职责

- 运行前端、后端、PostgreSQL、Qdrant、文档转换和备份容器。
- 保存案件文件、知识库原件和导入暂存文件。
- PostgreSQL 与 Qdrant 数据目录必须放 NAS 本机 SSD 卷，不放 SMB 网络挂载目录。
- NAS 容器只通过内部网络暴露 PostgreSQL、Qdrant；外部只开放 HTTPS 入口。

### 2.2 模型机职责

- 独立运行 LM Studio 或 llama.cpp，不与 NAS 数据库争抢内存和 I/O。
- 当前设备为 Windows 11、i9-14900KF、用户确认的 32GB 显存 GPU、64GB 内存；正式部署前以 `nvidia-smi`、LM Studio 模型页和实际加载结果复核显存、量化与上下文。
- 模型机运行生成模型、Embedding 模型和后续可选重排服务；生成与 Embedding 使用独立队列，避免批量索引挤占律师交互请求。
- ZGAI 继续使用单生成通道和队列；现有设备先按生成并发 1、Embedding 每批不超过 25 段、16K-32K 常用上下文试运行。
- 100K 上下文可作为特定长文任务能力，不作为所有请求默认值。长上下文会显著占用 KV Cache，并降低多人体验。

### 2.3 RAG 部署边界

- RAG 不是单一程序：知识原件在 NAS 文件卷，条目和权限在 PostgreSQL，向量在 Qdrant，Embedding/重排在模型机。
- 默认将 Qdrant 放在 NAS SSD 持久卷，模型机只通过局域网生成向量。这样模型机重启、切换模型或更新驱动时，案件、客户和知识检索底座仍可用，并可自动降级为关键词检索。
- 如果 NAS 的 CPU、内存或 SSD 性能使向量检索达不到目标，可将 Qdrant 单独部署到 Windows 的 WSL2/Docker 和本地 NVMe；必须限制 `6333/6334` 仅允许 ZGAI 后端访问，并定时把 collection 快照备份到 NAS。
- 不在模型机保存知识正文、案件原件或第二套 PostgreSQL；Qdrant payload 只保留必要引用与权限过滤字段。
- 不建议把整个 ZGAI 长期部署到模型机。Windows 更新、GPU 驱动、模型加载和高负载会形成业务与 AI 的共同单点，且不利于 7x24 小时文件、审批和数据库服务。

### 2.4 可选部署模式

| 模式 | 组件位置 | 适用阶段 | 结论 |
|---|---|---|---|
| A 推荐生产 | ZGAI/PostgreSQL/Qdrant/文件在 NAS；生成/Embedding 在模型机 | 正式试用与生产 | 默认采用，职责清晰、维护风险最低 |
| B 向量加速 | ZGAI/PostgreSQL/文件在 NAS；Qdrant/生成/Embedding 在模型机 | NAS 向量压测不达标 | 可选，要求本地 NVMe、快照回 NAS和服务自启动 |
| C 全部模型机 | 全部组件在 Windows 模型机 | 短期基准或 NAS 故障临时恢复 | 不作为长期生产方案 |
| D 腾讯云 Linux Demo | ZGAI/PostgreSQL/Qdrant/LibreOffice 在云端；文件与模型留在律所 | Mac 需要持续开发、Demo 要稳定在线 | 可行，须先完成 VPN、HTTPS、备份和断线测试 |

### 2.5 腾讯云 Linux Demo 替代方案

腾讯云 Linux 可以替代 Mac 承载稳定 Demo，并且比云端 Windows 更适合当前 Java、Vue、PostgreSQL、Qdrant、Nginx 和 LibreOffice 容器栈。推荐保持三层职责：

- 腾讯云 Linux：ZGAI 前后端、PostgreSQL、Qdrant、Nginx/HTTPS、LibreOffice 和定时数据库备份。
- 律所 Windows 模型机：LM Studio 生成与 Embedding，不保存业务主数据和案件原件。
- 律所 NAS：案件原件、知识原件和异地备份副本。

云端与律所内网之间必须建立 VPN 或等效的加密私网连接。公网安全组只开放 `443`，运维 SSH 仅允许固定来源或经 VPN 进入；`445`、`1234`、`5432`、`6333/6334` 不得暴露公网。由于云端读取 NAS 会受办公网络和跨网延迟影响，Demo 阶段可以先采用受控 VPN 文件访问；正式生产应评估 HTTPS 对象网关、MinIO 或加密同步，避免让 SMB 成为跨公网主链路。

轻量应用服务器适合人数较少、成本敏感、只使用脱敏数据且不强依赖律所内网资源的 Demo，并可直接使用 Linux 或 Docker 镜像。若需要腾讯云 VPC 与律所侧建立正式站点到站点 VPN、持续访问 NAS/模型机、接入更多云内服务或复杂网络策略，优先改用 CVM/VPC 架构；轻量服务器默认不与标准 VPC 资源互通，额外互联需要通过云联网设计。

试用规格不应低于 4 核 8GB；若 PostgreSQL、Qdrant、LibreOffice 与后端同机常驻，建议 8 核 16GB、100GB 以上 SSD，并给数据库、向量库和文档转换分别设置资源上限。轻量套餐、流量包和促销价格变化频繁，采购时以控制台同地域实时价格为准，不在 PRD 固化促销价。

仓库中的 `deploy/cloud` 已实现两种互斥模式、安全配置预检、回环端口绑定、Nginx HTTPS 模板、PostgreSQL 备份/恢复演练包装和 NAS 异地副本。部署前仍须在目标 Linux 上完成 Docker 镜像构建、VPN、NAS 挂载、域名证书和真实恢复验收。

选择顺序：当前局域网验收继续使用 Mac 稳定 Demo；确认云端规格、域名、备案和 VPN 后迁移至腾讯云 Linux；不选择云端 Windows 工作站，除非后续出现 Microsoft Office COM、Windows 专属中间件或云端 GPU 模型等明确依赖。

- 轻量服务器内网与云联网：https://cloud.tencent.com/document/product/1207/56847
- VPC 与本地数据中心 VPN 连接：https://cloud.tencent.com/document/product/215/36699

### 2.6 Word/Excel 生成

- 当前后端已使用 Apache POI，可原生生成 `.docx`、`.xlsx`，不要求 Microsoft Office。
- 增加 LibreOffice headless 容器，用于 `.docx/.xlsx -> PDF`、预览和打印稿转换。
- 容器安装经授权的中文字体，并固定 LibreOffice 与字体版本，避免不同电脑排版不一致。
- 正式模板保留“模板版本、生成参数、生成时间、案件、操作者”审计信息。
- 不采用 macOS Word 自动化作为生产方案；它难以容器化、并发控制和无人值守恢复。

## 3. AI 模型与路由

### 3.1 模型名称核验

模型机 `/v1/models` 已核验实际模型 ID 为 `qwen/qwen3.6-35b-a3b`，同时已加载 `text-embedding-nomic-embed-text-v1.5`，实测输出向量维度为 768。系统部署配置必须使用实际 ID 和维度，不用近似名称替代；量化等级、上下文上限和吞吐仍需在正式压测时记录。

### 3.2 路由规则

| 场景 | 默认模型 | 是否可切云端 | 规则 |
|---|---|---|---|
| 案件材料问答、归案、摘要 | 本地 Qwen | 可，须人工明确选择 | 选择云端前提示材料将发送给供应商；优先脱敏 |
| 日程/待办结构化指令 | 本地 Qwen | 可 | 最终写入仍遵守风险分级和确认规则 |
| 全所法规/制度问答 | 本地 Qwen + RAG | 可 | 返回知识来源；不得伪造引用 |
| 长文复杂推理 | 本地优先 | 可选 DeepSeek/GLM/Kimi | 用户手动选择，不自动把私密材料转云端 |
| 模型健康检查 | 本地 | 否 | 失败只提示切换，不静默转发案件材料 |

### 3.3 公用 Token 设计

- Token 只保存在 NAS Secret/环境变量或服务端加密配置中，禁止写入 Git、前端包、浏览器存储和普通日志。
- 管理员可配置、测试、轮换和停用；普通用户只看到“已配置”和可选模型，不看到密钥。
- 每次调用记录用户、案件、供应商、模型、输入/输出 token、耗时、状态和估算费用，不记录完整私密提示词。
- 分供应商设置月度预算、单用户日限额、最大上下文和最大输出；达到阈值后停止云调用并提示管理员。
- 云端默认不自动回退。只有用户主动选定云模型后，后端才向相应供应商发送内容。

## 6. 设备与一次性预算

### 6.1 生产 NAS 最低条件

| 项目 | 最低试用 | 推荐生产 |
|---|---|---|
| CPU | x86_64 4 核，支持容器 | x86_64 8 核以上 |
| 内存 | 16GB | 32GB ECC，数据量大时 64GB |
| 数据库/向量盘 | 1 块 SSD（仅试用） | 2 x 1.92TB 企业 SSD，RAID1 |
| 案件文件盘 | 现有冗余阵列 | RAID6/SHR-2，容量按 3 年增长预留 2 倍 |
| 网络 | 1GbE | 2.5GbE 起，批量文档建议 10GbE |
| 电源 | 无 | 在线/线路互动 UPS + 自动关机 |
| 备份 | 同机副本 | 独立卷 + 外置/异地加密备份 |

在确认 NAS 型号、CPU 架构、可扩展内存、SSD 插槽和 Container Manager 支持前，不应直接购买升级件。

### 6.2 规划预算区间

| 方案 | 一次性预算 | 适用结论 |
|---|---:|---|
| 沿用现有 NAS + 现有模型机 | CNY 0-8,000 | 优先方案；预算用于 SSD、内存、UPS 和备份盘 |
| 采购/升级生产 NAS（不含大容量 HDD） | CNY 12,000-30,000 | 现有 NAS 不支持 x86 容器、32GB 内存或 SSD 数据卷时采用 |
| 保留现有 32GB 显存/64GB 内存模型机 | CNY 0 | 先以并发 1 完成质量与吞吐基准 |
| 新购高内存 Mac Studio 模型机 | CNY 30,000-60,000+ | 需要更大上下文或更稳定并发后再定；Apple 官网基础款当前 CNY 16,499 起，实际需选择高内存配置 |
| 云 API 首期充值 | CNY 600 | DeepSeek、GLM、Kimi 各 200 |

上述硬件区间是采购预留，不是厂商正式报价。NAS 型号和 Mac 内存配置确定后，应取得含税正式报价。Apple 官网当前 Mac Studio 基础款价格不能代表满足本项目内存要求的最终价格。

## 7. 推荐资源分配

| 容器 | CPU 建议 | 内存建议 | 持久化位置 |
|---|---:|---:|---|
| PostgreSQL | 2-4 核 | 6-10GB | NAS SSD RAID1 |
| Qdrant | 2-4 核 | 6-10GB | NAS SSD RAID1 |
| ZGAI 后端 | 2-4 核 | 4-8GB | 日志与临时区 |
| ZGAI 前端/Nginx | 1 核 | 512MB-1GB | 配置与证书 |
| LibreOffice headless | 2 核 | 3-4GB | 临时转换区 |
| 备份任务 | 1 核 | 1GB | 独立备份卷 |

容器资源可以共享，但 PostgreSQL、Qdrant 和 LibreOffice 应设内存上限，避免单次大文件转换拖垮全部服务。

模型机建议预留至少 16GB 系统内存给 Windows、LM Studio 和缓存，不在生成模型占满内存时并行执行全库重建。Embedding 增量索引可常态运行，全库重建安排在夜间，并由 ZGAI 队列限速。

## 8. 落地顺序与验收

1. 读取 NAS 型号、CPU、内存、卷类型、SSD 和容器能力，出具设备适配结论。
2. 建立 NAS staging 环境，运行 PostgreSQL、Qdrant、ZGAI、LibreOffice 和备份容器。
3. 使用脱敏数据完成 H2 到 PostgreSQL 迁移、回滚和恢复演练。
4. 使用 `ZGAI旧案资料导入模板_v1.xlsx` 收集首批 20 个旧案，先预检再试导入。
5. 在模型机加载生成与 Embedding 模型，记录实际模型 ID、向量维度、显存/内存峰值、首 token 延迟和吞吐。
6. 在 NAS 启动 Qdrant，完成 1万、10万和目标规模向量压测；只有未达标时才试验模式 B，并比较迁移收益。
7. 使用 `scripts/check-rag-stack.sh` 核验模型 ID、Embedding 输出维度和 Qdrant collection 维度一致后，再启动知识索引；未配置模型时系统保持关键词模式且不连接 Qdrant。
8. 接入 AI 网关：本地默认，DeepSeek/GLM/Kimi 管理员配置，用户手动切换云端。
9. 建立 40 条指令、30 份文书、20 题知识问答评价集，记录准确率、延迟和费用。
10. 使用 `deploy/nas/backup-postgres.sh` 生成 custom-format 备份并记录 SHA-256，再用 `restore-drill.sh` 创建临时数据库完成离线恢复演练；快照不能替代逻辑备份与恢复验证。
11. 完成 NAS 断网、模型离线、Qdrant 离线/维度错误、云额度耗尽、文件转换失败和备份恢复测试后，才进入真实试用。

## 9. 尚需现场采集

- NAS 品牌型号、CPU 架构、当前/最大内存、DSM/系统版本、Container Manager 版本。
- 存储池、卷格式、SSD/HDD 型号与可用容量、快照和 UPS 状态。
- 模型机 `/v1/models` 返回的实际模型 ID、量化等级、上下文和吞吐。
- 预计同时在线人数、日均 AI 次数、单案文件量和旧案总量。
