> 版本：v2.1.0  
> 更新日期：2026-07-14  
> 当前阶段：核心业务闭环优化阶段  
> 当前重点：案件管理、客户管理、组织身份与权限基础、立案审批与案卷归档

---

## 1. 项目定位

ZGAI 是面向律所内部使用的案件与客户管理系统，目标不是先做“大而全”的工具集合，而是先把律师办案的核心链路做稳：

- 客户进入系统；
- 承办人发起案件；
- 行政管理完成利冲审查与立案审批；
- 审批通过后形成一案一档；
- 案件办理过程中的文档、记录、时间线、客户关系可持续沉淀；
- 后续再逐步接入 AI、财务、统计、外部工具等增强模块。

当前研发方向已经从“功能铺开”调整为“核心业务闭环优先、界面简化、权限分层、逐步扩展”。

### 2.2 身份类别

账号身份类别用于后续权限控制、审批流配置和数据可见范围控制：

| 身份类别 | 初步定位 |
|----------|----------|
| 主任 | 全所管理、全局数据、重大审批 |
| 合伙人 | 部门或团队管理、案件主办、审批参与 |
| 行政管理 | 立案审批、利冲审查、案卷建档、流程监管 |
| 律师 | 案件主办、案件办理、客户维护 |
| 实习律师 | 协助办案、材料录入、受限查看 |
| 助理 | 案件辅助、文档整理、流程协作 |

### 2.3 权限建设方向

当前已经建立“部门 + 身份类别”的账号基础。下一步应将该基础转化为权限矩阵：

| 权限对象 | 主任 | 合伙人 | 行政管理 | 律师 | 实习律师 | 助理 |
|----------|------|--------|----------|------|----------|------|
| 全所案件查看 | 是 | 可配置 | 否 | 否 | 否 | 否 |
| 本部门案件查看 | 是 | 是 | 可配置 | 可配置 | 可配置 | 可配置 |
| 本人承办案件 | 是 | 是 | 可配置 | 是 | 协作范围 | 协作范围 |
| 客户新增/编辑 | 是 | 是 | 可配置 | 是 | 受限 | 受限 |
| 立案审批 | 可监管 | 可参与 | 是 | 否 | 否 | 否 |
| 案卷归档 | 可监管 | 可查看 | 是 | 协作 | 协作 | 协作 |
| 系统设置 | 是 | 否 | 可配置 | 否 | 否 | 否 |

## 3. 当前系统架构

### 3.1 技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| 前端 | Vue 3 + Vite + Element Plus | 当前开发端口 `3017` |
| 后端 | Spring Boot 2.7 + Java 11 | API 前缀 `/api`，当前端口 `8080` |
| 数据库 | H2 文件模式 | 当前用于本地开发验证，后续建议迁移 MySQL/PostgreSQL |
| 认证 | JWT + Spring Security | 已具备登录认证基础 |
| 文件 | 本地文件系统 | 已开始按案件文件夹归档 |
| AI/知识库 | DeepSeek、Qdrant、DashScope 等 | 暂作为增强模块，核心流程阶段不作为主线 |

### 3.2 当前界面策略

已将主界面方向调整为类 macOS 的简洁专业风格，导航先保留核心模块：

- 总览；
- 案件管理；
- 客户管理。

其他模块如 AI、财务、统计、行政 OA、工具集等暂不作为主导航重点，后续按成熟度逐步开放。

## 6. 当前问题与风险

| 问题 | 影响 | 建议 |
|------|------|------|
| 权限仍偏粗 | 不同身份的数据边界尚未完全隔离 | 下一阶段建立权限矩阵与接口级控制 |
| 审批 UI 仍偏通用 | 行政管理处理立案审批时不够贴合利冲审查场景 | 增加专用“立案审批工作台” |
| 利冲报告为基础文本版 | 目前可留档，但专业性与模板化不足 | 做成结构化表单 + PDF/Word 输出 |
| 案卷目录结构初版较简单 | 后续文件多时需要分类目录 | 建立标准目录：立案审批、委托材料、证据材料、法律文书、庭审材料、结案归档 |
| 当前数据库仍为 H2 | 多用户正式使用风险较高 | 迁移 MySQL/PostgreSQL |
| 历史模块较多 | 导航与权限未收敛时会影响用户理解 | 继续隐藏非核心模块，成熟一个开放一个 |
| 自动化测试不足 | 回归风险较高 | 增加核心链路 E2E 测试 |

## 7. 下一阶段研发路线

### 阶段一：核心闭环稳定化（建议 1-2 周）

目标：让案件与客户系统可被真实试用。

重点任务：

| 任务 | 说明 |
|------|------|
| 完善立案审批工作台 | 行政管理可集中查看待审批案件 |
| 利冲审查结构化 | 增加委托方、相对方、关联主体、审查结论、审查人、审查时间 |
| 利冲报告模板化 | 支持生成 PDF/Word 留档 |
| 案件文件夹标准化 | 审批通过后生成标准目录 |
| 案件列表状态优化 | 待审批、驳回、审理中等状态更清晰 |
| 客户详情增强 | 增加客户来源、所属部门、关联案件统计、沟通记录 |
| 表单校验统一 | 电话、证件号、统一社会信用代码、金额等规则统一 |

### 阶段二：权限与数据边界（建议 2-3 周）

目标：让不同身份的人看到不同数据、做不同操作。

重点任务：

| 任务 | 说明 |
|------|------|
| 权限矩阵落库 | 部门 + 身份类别 + 角色权限 |
| 案件数据权限 | 主任看全所，合伙人看部门/团队，律师看本人/参与案件 |
| 客户数据权限 | 客户归属部门、负责人、协作人控制 |
| 操作权限控制 | 新建、编辑、删除、归档、审批、导出分别控制 |
| 审批权限控制 | 只有行政管理可处理立案审批 |
| 审计日志强化 | 关键操作留痕：审批、归档、删除、导出 |

### 阶段三：专业化办案体验（建议 3-5 周）

目标：让系统从“能用”变为“好用”。

重点任务：

| 任务 | 说明 |
|------|------|
| 案件详情重构 | 围绕办案过程重排信息架构 |
| 办案时间线增强 | 自动记录审批、文件上传、状态变更 |
| 文档中心重构 | 按案卷目录树管理文件 |
| 常用文书模板 | 委托合同、授权委托书、风险告知、利冲报告 |
| 案由索引库 | 建立可维护的案由/罪名/非诉事项参考库 |
| 搜索能力 | 支持客户、案件、当事人、案由、案号统一搜索 |

### 阶段四：智能化与经营分析（后续增强）

目标：在核心数据稳定后释放 AI 和统计价值。

重点任务：

| 任务 | 说明 |
|------|------|
| AI 辅助录案 | 从委托材料或起诉状提取案件信息 |
| AI 文书生成 | 基于案件数据生成初稿 |
| 类案检索优化 | 围绕案由、法院、争议焦点、裁判观点 |
| 经营统计 | 部门案件数、收费、回款、结案率 |
| 风险预警 | 审限、开庭、收费节点、长期未推进案件 |

## 8. 近期验收标准

### 8.1 案件管理验收

- 新建案件页面无“案件等级”；
- 案由为自由填写；
- 收费方式仅显示四项：固定收费、风险收费、基础+风险、其他；
- 提交后案件进入“待审批”；
- 案件列表可筛选并展示“待审批”；
- 审批通过后案件进入“审理中”；
- 审批驳回后案件进入“立案驳回”；
- 审批通过后自动生成案件文件夹与利冲审查报告。

### 8.2 客户管理验收

- 新建客户无“客户等级”；
- 客户可选择所属部门；
- 手机号和区号+固话均可通过校验；
- 客户详情能看到关联案件；
- 客户列表、详情、编辑均调用真实后端数据。

### 8.3 权限基础验收

- 用户可维护所属部门；
- 用户可维护身份类别；
- 行政管理账号可收到立案审批；
- 非行政管理账号后续需限制审批入口。

## 9. 结论

ZGAI 当前已经具备案件管理、客户管理、组织身份、立案审批、案卷建档的基础闭环。下一阶段不宜继续横向扩张模块，而应围绕真实律所办案流程继续做深：

1. 做稳案件与客户；
2. 做清审批与权限；
3. 做好案卷与文档；
4. 最后再叠加 AI、统计、工具集等能力。

该路线能降低系统复杂度，让领导和一线使用者更容易看到阶段成果，也更适合逐步试点上线。
