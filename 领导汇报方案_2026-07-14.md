# ZGAI 系统开发进展与下一步研发优化方案

> 汇报日期：2026-07-14  
> 汇报主题：案件管理与客户管理核心闭环建设进展  
> 汇报对象：律所管理层 / 项目负责人

---

## 一、汇报摘要

ZGAI 系统近期研发方向已从“多模块全面铺开”调整为“围绕律师真实办案流程，优先完善案件管理与客户管理核心闭环”。

目前系统已经完成客户管理、案件管理、部门身份基础、立案审批、案卷建档等关键能力的阶段性优化。新建案件不再直接进入普通案件列表，而是由承办人发起立案申请，行政管理进行利冲审查与审批，审批通过后自动形成案件档案文件夹，并生成利冲审查报告留档。

这意味着系统已经从“信息录入工具”开始转向“律所办案流程管理系统”。

---

## 二、当前阶段已完成工作

### 1. 客户管理优化

已完成内容：

- 取消客户“客户等级”字段；
- 改为客户“所属部门”；
- 部门结构已按律所实际组织调整为九大部门；
- 联系电话规则已支持手机号和区号+固话；
- 客户列表、客户详情、新建客户、编辑客户已接入真实后端数据；
- 客户详情可展示关联案件。

当前价值：

- 客户不再按简单等级划分，而是进入部门化管理；
- 为后续部门数据统计、客户归属、权限控制打基础；
- 更符合律所内部管理方式。

### 2. 账号组织架构优化

已完成内容：

- 账号支持“所属部门”；
- 账号支持“身份类别”；
- 身份类别包括：主任、合伙人、行政管理、律师、实习律师、助理；
- 默认行政审批账号已可配置为“行政管理”。

当前价值：

- 为后续不同身份设置不同权限提供基础；
- 可以支持主任、合伙人、律师、行政、助理等不同使用场景；
- 为案件审批、数据可见范围、操作权限控制提供基础字段。

### 3. 案件创建流程优化

已完成内容：

- 新建案件取消“案件等级”；
- 案由从下拉选择改为自由填写；
- 按案件类型提供参考案由/罪名索引提示；
- 收费方式调整为四类：固定收费、风险收费、基础+风险、其他；
- 新建案件主按钮调整为“提交立案申请”。

当前价值：

- 案由录入更贴近真实办案；
- 避免固定选项限制律师表达；
- 收费方式更符合律所常见收费模式；
- 流程语言更清晰，避免“新建案件”“确认立案”“提交审批”混用。

### 4. 立案审批流程打通

已完成内容：

- 承办人提交立案申请后，案件状态显示为“待审批”；
- 系统自动生成“立案审批”审批单；
- 审批流转至身份类别为“行政管理”的人员；
- 行政管理审批通过后，案件状态变为“审理中”；
- 行政管理驳回后，案件状态变为“立案驳回”；
- 审批通过后自动创建案件文件夹；
- 自动生成利冲审查报告并登记为案件文档。

当前价值：

- 建立了正式立案前的内部审查机制；
- 将利冲审查从线下人工动作转为系统留痕；
- 案件审批过程可追踪；
- 审批通过后才正式建档，符合律所规范管理要求。

### 5. 案件文档与案卷基础

已完成内容：

- 案件审批通过后自动建立对应案卷文件夹；
- 利冲审查报告进入案卷文件夹；
- 后续案件文件上传会归入对应案件文件夹；
- 待审批案件限制提前上传正式案件文件。

当前价值：

- 初步实现“一案一档”；
- 后续可继续扩展为标准案卷目录；
- 为案件材料归档、文档检索、结案归档打基础。

### 6. UI 与导航方向调整

已完成内容：

- 主导航已简化；
- 当前优先保留总览、案件管理、客户管理；
- 界面方向调整为类 macOS 的简洁专业风格；
- 暂时弱化 AI、财务、统计、工具等非核心入口。

当前价值：

- 降低系统复杂度；
- 让使用者先聚焦核心业务；
- 便于后续成熟一个模块、开放一个模块。

## 四、当前系统价值

### 1. 管理价值

- 案件进入系统前增加审批控制；
- 行政管理可介入利冲审查；
- 案件是否正式建立有明确状态；
- 案件资料有统一归档位置；
- 未来可按部门、人员、案件状态进行统计。

### 2. 律师使用价值

- 承办人可快速发起立案申请；
- 案由、收费方式等字段更符合真实业务；
- 案件信息、客户信息、办案记录、文档逐步集中管理；
- 减少案件材料散落在个人电脑或聊天记录中的情况。

### 3. 行政使用价值

- 可集中处理立案审批；
- 利冲审查可形成报告；
- 案卷建立有系统留痕；
- 为后续档案管理、归档检查提供依据。

## 五、当前不足与风险

| 问题 | 当前影响 | 建议 |
|------|----------|------|
| 权限体系仍是基础版 | 不同身份数据边界尚未完全隔离 | 下一阶段优先做权限矩阵 |
| 审批页面仍偏通用 | 行政管理处理利冲审查不够专用 | 建设立案审批工作台 |
| 利冲报告格式较基础 | 可留档但专业展示不足 | 做结构化表单和模板报告 |
| 案卷目录结构较简单 | 文件多后不易管理 | 建立标准案卷目录 |
| 数据库仍是开发型 H2 | 不适合正式多用户长期使用 | 迁移 MySQL 或 PostgreSQL |
| 自动化测试不足 | 后续迭代容易产生回归 | 增加核心流程自动化测试 |
| 历史模块较多 | 过早开放会影响使用体验 | 继续隐藏非核心模块 |

## 六、下一步研发方向建议

### 阶段一：案件与客户闭环稳定化

建议周期：1-2 周

目标：形成可试用、可演示、可闭环的核心系统。

重点任务：

1. 建设行政管理“立案审批工作台”；
2. 将利冲审查做成结构化表单；
3. 生成正式版利冲审查报告；
4. 建立标准案卷目录；
5. 优化案件列表状态展示；
6. 强化客户详情、客户关联案件、沟通记录；
7. 统一电话、证件号、金额等表单校验。

预期成果：

- 案件从申请、审批、建档、办理形成完整闭环；
- 行政管理可以真实参与流程；
- 可作为第一版内部试点系统。

### 阶段二：权限与数据边界建设

建议周期：2-3 周

目标：让不同身份的人看到不同数据、做不同操作。

重点任务：

1. 建立主任、合伙人、行政管理、律师、实习律师、助理权限矩阵；
2. 案件数据按全所、部门、本人、协作范围控制；
3. 客户数据按部门、负责人、协作范围控制；
4. 审批权限限制为行政管理及授权人员；
5. 删除、归档、导出等敏感操作增加权限控制；
6. 增强操作日志和审批留痕。

预期成果：

- 支持多人真实使用；
- 防止数据越权查看；
- 满足律所内部管理要求。

### 阶段三：专业办案体验提升

建议周期：3-5 周

目标：让系统从“能用”升级为“好用”。

重点任务：

1. 重构案件详情页信息架构；
2. 增强办案时间线；
3. 建设文档目录树；
4. 增加常用文书模板；
5. 建立案由/罪名/非诉事项索引库；
6. 增加统一搜索能力。

预期成果：

- 律师办理案件时能在一个页面看到关键信息；
- 案件材料更加有序；
- 文书、记录、客户、费用、审批可以形成统一沉淀。

### 阶段四：智能化与经营分析

建议周期：核心系统稳定后逐步推进

目标：在核心数据稳定后释放 AI 与统计价值。

重点任务：

1. AI 辅助录案；
2. AI 文书初稿生成；
3. 类案检索优化；
4. 部门案件统计；
5. 收费与回款统计；
6. 审限、开庭、长期未推进案件预警。

预期成果：

- 提升律师办案效率；
- 为主任和合伙人提供经营管理数据；
- 逐步形成律所知识资产。

---

## 七、建议管理层决策事项

建议领导确认以下方向：

1. 是否同意系统现阶段聚焦“案件管理 + 客户管理”；
2. 是否同意暂缓开放 AI、财务、NPA、工具集等非核心模块；
3. 是否确认立案审批必须由行政管理完成；
4. 是否确认利冲审查报告为立案必备留档材料；
5. 是否确认一案一档作为案件文档管理标准；
6. 是否启动权限矩阵建设；
7. 是否安排 1-2 名行政管理人员参与试用反馈；
8. 是否安排 2-3 名律师进行真实案件录入试点。

---

## 八、阶段目标建议

### 近期目标

用 1-2 周完成“案件立案审批 + 客户归属 + 案卷建档”的稳定试用版。

### 中期目标

用 1 个月完成权限控制、立案审批工作台、利冲报告模板、案件文档目录树。

### 长期目标

在核心案件与客户数据稳定后，逐步扩展 AI 文书、经营统计、类案检索、知识库沉淀。

---

## 九、汇报结论

ZGAI 当前已经具备从客户管理、案件创建、立案审批、利冲审查、案卷建档到案件办理的核心基础。下一步不建议继续横向堆叠模块，而应围绕律所真实办案流程做深做稳。

建议项目进入“核心业务闭环试点阶段”，先以案件管理和客户管理为主线，完成内部试用，再逐步开放权限、文档、统计、AI 等增强能力。
