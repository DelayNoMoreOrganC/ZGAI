## 1. 阅读顺序

每个开发 Agent 开始前必须依次阅读：

1. [README.md](README.md)：启动、配置、测试和安全边界。
2. [PRD.md](PRD.md)：业务规则、范围和验收标准。
3. 本文：当前进度、文件边界和合并要求。

旧阶段报告、轮次总结和旧 Agent 指令已清理，不能根据历史聊天或旧结论判断完成度；以当前代码、测试和这三份文档为准。

## 2. 当前运行基线

| 项目 | 状态 |
|---|---|
| 前端 | Vue 3/Vite，端口 `3017`，生产构建通过 |
| 后端 | Spring Boot 2.7/Java 11，端口 `8080`，101 项测试通过 |
| 当前开发数据库 | H2 文件库 |
| 目标数据库 | PostgreSQL 16，profile 和初始化脚本已存在 |
| 案件文件 | 本机/NAS；当前 NAS 约定挂载点 `/Volumes/ZGAI` |
| 知识文件 | NAS `knowledge-files` 或配置目录 |
| RAG | 未配置向量/模型时为 `KEYWORD / RETRIEVAL_ONLY` |
| Qdrant | 代码已接入，当前运行环境未稳定启用 |
| 省时宝 | 当前直接跳转独立项目，深度融合暂缓 |

已知非阻断警告：

- H2 启动时可能出现历史重复索引提示，需在 PostgreSQL 迁移前清理初始化策略。
- 前端构建存在 Dart Sass legacy API 和大分块警告。
- 系统可用于局域网受控试用，不等于完成生产部署。

## 6. 推荐并行工作流

每个工作流使用独立分支和 worktree。分支统一使用 `codex/` 前缀。

| 工作流 | 推荐分支 | 主要目录 | 交付物 |
|---|---|---|---|
| A 案件与审批 | `codex/case-approval` | `controller/Case*`、`service/Case*`、`Approval*`、案件前端 | 两级审批 E2E、审批规则配置化 |
| B 客户与利冲 | `codex/client-conflict` | `Client*`、`Conflict*`、客户前端 | 主体关系、利冲报告、权限回归 |
| C 文档与数据库 | `codex/storage-postgres` | `CaseDocument*`、`Storage*`、SQL、备份脚本 | PostgreSQL 迁移、NAS/灾备测试 |
| D 知识与 AI | `codex/knowledge-rag` | `Knowledge*`、`RAG*`、`Qdrant*`、法律数据适配器、知识前端 | 内容导入、供应商对照评价、模型接入 |
| E 前端设计 | `codex/ui-system` | `frontend/src/layouts`、样式、共享组件 | 设计 token、四类工作台和响应式页面 |
| F QA 与安全 | `codex/qa-security` | `backend/src/test`、E2E、`scripts/` | 角色矩阵、浏览器回归、漏洞与压测报告 |

## 7. 文件所有权与冲突控制

以下文件是共享热点，同一时间只能由一个工作流修改：

- `backend/src/main/java/com/lawfirm/security/SecurityConfig.java`
- `backend/src/main/java/com/lawfirm/security/UserAuthorityService.java`
- `backend/src/main/java/com/lawfirm/config/DataInitializer.java`
- `backend/src/main/resources/application*.yml`
- `backend/src/main/resources/db/postgres/001_foundation_schema.sql`
- `frontend/src/router/index.js`
- `frontend/src/layouts/MainLayout.vue`
- `frontend/src/stores/user.js`
- `PRD.md`、`handoff.md`、`README.md`

修改共享热点前，Agent 必须在任务说明中声明；若另一分支正在修改，由集成人员统一处理，禁止复制覆盖。

跨模块契约要求：

- 新增权限码时同步后端权限矩阵、前端菜单/路由和角色测试。
- 新增数据库字段时同步实体、DTO、SQL、映射层、前端和迁移验证。
- 修改状态机时同步枚举、服务、显示文本、审计、通知和测试。
- 新增文件下载时必须复用业务权限，并校验根目录与真实路径。
- 新增 AI 数据源时必须明确公开范围、授权、时效和是否可索引。
- 商业法律库只能使用供应商正式 API/MCP 和企业凭据；网页会员、人工下载权限不得用于抓取、整库复制或全文向量化。

## 8. 分支与合并流程

1. 从最新 `master` 创建 `codex/<scope>` 分支和独立 worktree。
2. 一个分支只处理一个工作流，不混入无关重构。
3. 提交前运行本模块测试、后端全量测试和前端构建。
4. PR 描述列出数据库变更、权限影响、配置项、验证结果和回滚方式。
5. 合并顺序：数据模型/权限契约优先，业务服务其次，前端最后，QA 回归收尾。
6. 合并后由集成人员更新本文周进度、测试基线和剩余缺口。

禁止：

- 多个 Agent 直接在同一工作目录同时修改。
- 未同步主分支就覆盖共享热点。
- 用前端按钮隐藏代替后端权限。
- 提交真实密码、AI 密钥、客户数据或 NAS 凭据。
- 根据旧报告宣称功能已完成。
- 把 H2 测试成功当作 PostgreSQL 迁移成功。

## 9. 验证清单

### 每个 PR 必跑

```bash
cd backend
JAVA_HOME=/opt/homebrew/opt/openjdk@11 /opt/homebrew/opt/maven/bin/mvn test
```

```bash
cd frontend
npm run build
```

### 权限或流程变更

```bash
./scripts/smoke-test-core.sh
./scripts/smoke-test-roles.sh
```

### 知识与 AI 变更

```bash
./scripts/evaluate-knowledge-rag.sh
```

必须人工检查：

- 未登录返回 `401`。
- 无权限返回 `403` 或统一业务拒绝，不泄露对象存在性和详情。
- 普通律师看不到其他部门案件/客户。
- 利冲可全库命中但不能打开无权客户。
- 行政能看到审批资料并明确操作，主任能终审。
- 文件和备份接口不返回物理绝对路径。
- 案件材料不进入知识库或 RAG。

## 10. 数据与环境注意事项

- 当前局域网测试数据可能位于 H2 文件库，切换 PostgreSQL 前先备份。
- NAS 挂载点不能写死在业务代码，必须使用配置项。
- 测试旧资料扫描时只使用只读目录和脱敏样本。
- 不删除或重置用户已有数据，除非有明确备份和书面确认。
- `admin`/`amin` 保护逻辑用于开发管理，正式生产应迁移为受控超级管理员机制。

## 11. 等待用户输入的事项

- PostgreSQL 部署位置和运维方式。
- 四类真实测试账号的安全提供方式。
- 旧案资料根目录、脱敏样本和命名规则。
- 元典、北大法宝或其他外部法律库的试用账号、预算，以及 API、系统嵌入、缓存和 RAG 授权范围。
- 首批律所内部制度与公共模板文件。
- AI 模型基准测试后再决定硬件采购。

Agent 遇到上述事项应暂停对应分支，不应自行猜测；仍可继续不依赖该决策的测试、权限、数据模型和 UI 工作。

## 12. 交接完成标准

Agent 完成任务时必须提供：

- 变更文件与业务效果。
- 权限和数据范围影响。
- 数据库/配置变更与回滚方法。
- 自动测试和人工验证结果。
- 未验证风险与下一步。
- 对 `handoff.md` 的最小更新。

没有上述证据的“完成”只视为开发中，不进入主分支完成清单。
